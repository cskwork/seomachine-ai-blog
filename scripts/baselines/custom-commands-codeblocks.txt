```markdown
# .claude/commands/review.md

아래 변경된 파일에 대해 코드 리뷰를 수행해줘.

리뷰 대상: $ARGUMENTS

다음 네 가지 관점에서 분석하고, 각 이슈마다 심각도를 표시해줘:

**1. 버그 가능성**
- 엣지 케이스 누락 여부
- 널 참조, 배열 범위 초과, 타입 불일치 가능성
- 비동기 처리의 race condition 위험

**2. 보안 취약점**
- SQL 인젝션, XSS, CSRF 취약점
- 하드코딩된 비밀번호나 API 키
- 인증/인가 로직의 허점

**3. 성능 문제**
- 불필요한 재렌더링이나 반복 연산
- N+1 쿼리 패턴
- 메모리 누수 가능성

**4. 코딩 컨벤션 위반**
- 프로젝트 네이밍 컨벤션 준수 여부
- 함수 길이와 복잡도
- 중복 코드와 추상화 기회

심각도 기준:
- Critical: 즉시 수정 필요 (서비스 장애, 보안 침해 가능)
- High: 이번 PR에서 반드시 수정
- Medium: 다음 스프린트 내 수정 권장
- Low: 개선 제안 (선택 사항)

리뷰가 끝나면 전체 요약과 함께 "수정 없이 머지 가능 여부"를 판단해줘.
```
```bash
# 특정 파일 리뷰
/review src/auth/login.ts

# 여러 파일 리뷰
/review src/auth/login.ts src/auth/middleware.ts

# 변경된 파일 전체 리뷰 (git diff 기반)
/review $(git diff --name-only HEAD~1)
```

```markdown
# .claude/commands/test-feature.md

기능명: $ARGUMENTS

위 기능과 관련된 테스트를 전방위적으로 점검해줘. 다음 순서로 진행해:

## 1단계: 관련 파일 탐색

다음 패턴으로 관련 파일을 찾아줘:
- 소스 파일: `src/**/*$ARGUMENTS*`, `src/**/$ARGUMENTS/**`
- 테스트 파일: `tests/**/*$ARGUMENTS*`, `**/__tests__/*$ARGUMENTS*`, `**/*.test.*`, `**/*.spec.*`

찾은 파일 목록을 정리해서 보여줘.

## 2단계: 테스트 실행

발견된 테스트 파일에 대해 실행 명령을 제안해줘:
- Jest 프로젝트: `npx jest --testPathPattern="$ARGUMENTS" --coverage`
- Vitest 프로젝트: `npx vitest run --reporter=verbose $ARGUMENTS`
- pytest 프로젝트: `pytest tests/ -k "$ARGUMENTS" -v --cov`

프로젝트의 package.json이나 pyproject.toml을 확인해서 적합한 명령을 골라줘.

## 3단계: 커버리지 분석

현재 테스트가 다루고 있는 케이스와 다루지 않는 케이스를 분류해줘:

**커버되는 케이스:**
- (현재 테스트에서 검증하는 시나리오 목록)

**커버되지 않는 케이스:**
- (누락된 엣지 케이스, 오류 시나리오, 경계값)

## 4단계: 누락 테스트 케이스 제안

커버되지 않는 케이스에 대해 구체적인 테스트 코드를 작성해줘. 프로젝트에서 사용 중인 테스트 프레임워크 스타일로 맞춰줘.

각 테스트에는 다음을 포함해:
- 명확한 테스트 설명 (it/test 블록의 설명)
- Arrange-Act-Assert 구조
- 예상 결과값
```
```bash
# 로그인 기능 테스트 점검
/test-feature login

# 결제 모듈 테스트 점검
/test-feature payment

# 사용자 프로필 기능 테스트 점검
/test-feature user-profile
```

```markdown
# .claude/commands/deploy-check.md

배포 환경: $ARGUMENTS

배포 전 최종 점검을 수행해줘. 각 항목을 순서대로 확인하고 결과를 체크리스트 형태로 보고해줘.

## 체크리스트

### 빌드 및 컴파일
- [ ] 빌드 명령이 오류 없이 완료되는지 확인
  - 확인 명령: `npm run build` 또는 `yarn build`
  - 오류가 있다면 오류 내용과 수정 방법을 알려줘

### 테스트 통과
- [ ] 전체 테스트 스위트가 통과하는지 확인
  - 확인 명령: `npm test` 또는 `yarn test`
  - 실패한 테스트가 있다면 목록과 원인을 알려줘

### 타입 안전성
- [ ] TypeScript 타입 오류가 없는지 확인
  - 확인 명령: `npx tsc --noEmit`
  - 타입 오류가 있다면 위치와 수정 방법을 알려줘

### 코드 품질
- [ ] 린트 오류가 없는지 확인
  - 확인 명령: `npm run lint` 또는 `npx eslint src/`
  - 오류가 있다면 자동 수정 가능 여부와 수동 수정 필요 항목을 구분해줘

### 환경 변수
- [ ] 필요한 환경 변수가 모두 설정되어 있는지 확인
  - `.env.example` 또는 환경 변수 문서를 참고해서 누락된 변수를 찾아줘
  - 프로덕션 환경($ARGUMENTS)에 맞는 값이 설정됐는지 확인

### 의존성
- [ ] `package.json`의 의존성과 `node_modules`가 동기화되어 있는지 확인
  - `package-lock.json`이나 `yarn.lock`의 변경사항이 커밋됐는지 확인

### 보안
- [ ] 하드코딩된 비밀값이 없는지 확인
  - API 키, 비밀번호, 토큰이 소스 코드에 직접 포함되지 않았는지 점검
  - `.env` 파일이 `.gitignore`에 포함되어 있는지 확인

## 최종 판정

모든 항목 점검 후 다음 중 하나로 결론 내려줘:
- **배포 가능**: 모든 항목이 통과됨
- **조건부 배포 가능**: Minor 이슈가 있으나 배포 차단 수준이 아님 (이슈 목록 포함)
- **배포 불가**: 반드시 수정이 필요한 Critical 이슈가 있음 (이슈 목록과 수정 우선순위 포함)
```
```bash
# 프로덕션 배포 전 점검
/deploy-check production

# 스테이징 배포 전 점검
/deploy-check staging
```
