## Git 커밋 그래프가 말해주는 변화

디자이너의 Git 커밋 활동이 갑자기 늘어난다면, 무슨 일이 벌어진 걸까.

Jeff Zych는 자신의 Git 커밋 그래프에서 뚜렷한 변곡점을 발견했습니다. 활동량이 급증한 시점은 Claude Code를 일상 도구로 쓰기 시작한 때와 정확히 일치했습니다. 동시에 Figma 사용량은 눈에 띄게 줄었다.

> 이 글은 Jeff Zych의 ["Claude Code is Re-shaping My Design Process"](https://jlzych.com/2026/02/20/claude-code-is-re-shaping-my-design-process/) 원문을 바탕으로 한국어로 재구성했습니다.

단순히 도구가 바뀐 것이 아니다. 디자인 프로세스 자체의 구조가 달라졌습니다.

---

## 핵심 변화: 중간 단계를 건너뛴다

Zych가 설명하는 변화의 핵심은 **중간 피델리티 목업의 소멸**입니다.

> "지금은 중간 피델리티 목업을 완전히 건너뛴다. Claude Code로 바로 가서, 적절한 프롬프트만 주면 수 분 만에 꽤 괜찮은 버전이 작동합니다."

기존 워크플로우에서는 Figma 와이어프레임 > 중간 목업 > 핸드오프 > 개발자 빌드 순서를 따랐다. 이제는 문제 정의 > Claude Code로 직접 빌드 > 반복 수정 > PR 제출로 흐름이 단축됩니다.

### 기존 워크플로우 vs 현재

| 단계 | 기존 | 현재 |
|------|------|------|
| 1 | Figma에서 와이어프레임 | 문제와 사용자 경험 서술 |
| 2 | 중간 피델리티 목업 | Claude Code로 직접 빌드 |
| 3 | 개발자에게 핸드오프 | 프롬프트로 반복 수정 |
| 4 | 개발자 빌드 + 리뷰 루프 | 작동하는 결과물에서 폴리싱 |
| 5 | 몇 주 후 완성 | 수 분~수 시간 내 완성 |

---

## 작업 규모에 따라 달라지는 접근법

모든 작업에 같은 방식을 적용하지는 않습니다. Zych는 작업 규모에 따라 도구 선택을 구분합니다.

### 소규모 기능 개선

Figma를 아예 열지 않습니다. 와이어프레임도, 핸드오프도 없습니다. Claude Code에서 바로 구현하고 끝냅니다. 사용성 개선이나 이미 확립된 패턴의 반복 적용에 적합하다.

### 대규모 기능 개발

탐색 단계는 여전히 필요하다. 다만 방식이 다르다. 문제를 서술하고, 사용 사례와 예상 동작을 정리한 뒤, Claude 채팅에서 아이디어를 탐색합니다. 이후 인터랙티브 아티팩트를 만들어 팀 리뷰와 피드백을 받습니다.

### 시각 탐색

색상 조합, 타이포그래피 옵션, 새로운 컴포넌트를 비교하는 작업에서는 여전히 Figma가 낫다. 여러 대안을 나란히 놓고 비교하기에 캔버스가 더 직관적이기 때문입니다.

### 최종 구현

상태 관리, 인터랙션 튜닝, 마감 폴리싱은 Claude Code에서 처리한 뒤 PR을 제출합니다.

---

## "내가 통제하는 엔지니어"

Zych의 표현 중 가장 인상적인 대목입니다.

> "Claude Code로 의도한 사용자 경험을 서술한다 -- 사용 사례, 예상 동작, 기능이 어떻게 발전할지까지. 이건 원래 내가 엔지니어와 협업하던 방식과 똑같다. 다만 이번에는 그 엔지니어를 내가 직접 제어하고, 내 말대로 움직입니다."

디자이너가 의도를 설명하고 엔지니어가 해석하는 기존 구조에서, 디자이너가 의도를 설명하면 AI가 즉시 구현하는 구조로 전환된 것입니다. 해석 과정에서 손실되던 디자인 의도가 그대로 유지됩니다.

### 실제 데이터로 작업하는 이점

> "작동하는 버전에서 실제 데이터를 보면서 직접 수정하면, 전체 UX를 개선하는 사소한 것들이 바로 눈에 들어옵니다."

목업에서는 가짜 데이터로 작업합니다. 실제 작동하는 인터페이스에서는 진짜 데이터가 흐르기 때문에, 에지 케이스와 개선 포인트가 자연스럽게 드러난다.

---

## Opus 4.6의 코드베이스 이해력

Zych는 특히 Opus 4.6의 능력에 주목합니다.

> "Opus 4.6은 코드베이스를 이해하는 능력이 놀랍다. 명시적으로 지시하지 않아도 올바른 방향으로 작업합니다."

기존 디자인 시스템이 확립된 프로덕트에서 작업할 때, 모델이 기존 패턴과 컴포넌트를 파악하고 일관된 코드를 생성한다는 의미다. 디자이너가 매번 기술 스펙을 상세히 지정하지 않아도 된다는 점이 핵심 가치다.

---

## 바뀌지 않은 것

도구가 바뀌었다고 디자인의 본질까지 바뀌지는 않습니다. Zych는 이 점을 분명히 합니다.

> "바뀌지 않은 것은 사용자에게 집중하고, 그들의 니즈를 이해하고, 문제를 파악하고, 다양한 선택지를 탐색하고, 반복하고, 테스트하고, 다듬는 것입니다."

실행 속도가 빨라졌을 뿐, 디자인 사고의 프레임워크 자체는 그대로다.

| 변한 것 | 변하지 않은 것 |
|---------|--------------|
| 도구 (Figma 비중 감소) | 사용자 중심 사고 |
| 속도 (수 분 단위 프로토타입) | 문제 이해와 정의 |
| 프로세스 (중간 단계 생략) | 다양한 대안 탐색 |
| 핸드오프 (직접 구현) | 반복과 폴리싱 |

---

## Figma의 남은 역할

Figma가 사라지지는 않습니다. 역할이 좁아지고 전문화될 뿐입니다.

> "Figma는 여전히 시각 탐색에서 이긴다. 색상 조합, 타이포그래피 옵션, 새 컴포넌트 -- 이런 것들은 나란히 비교하기가 훨씬 쉽다."

이미 디자인 시스템이 갖춰진 기존 프로덕트에서 작업할 경우, Figma가 담당하는 영역은 **프로세스의 더 작은 부분**이 됩니다. 반면 완전히 새로운 프로덕트나 대규모 비주얼 리디자인에서는 Figma의 캔버스가 여전히 강력하다.

---

## 디자이너에게 주는 시사점

Zych의 경험에서 끌어낼 수 있는 실전 교훈은 다음과 같다.

### 1. 프롬프팅은 디자인 커뮤니케이션의 확장이다

엔지니어에게 의도를 설명하는 능력이 곧 Claude Code에 프롬프트를 작성하는 능력입니다. 디자인 의도를 명확히 서술하는 훈련이 가장 중요하다.

### 2. 실제 작동 환경에서 디자인 결정을 내린다

목업이 아닌 실제 데이터가 흐르는 인터페이스에서 판단하면, 결정의 품질이 올라갑니다.

### 3. 도구 전환이 아니라 프로세스 재구성이다

Figma를 Claude Code로 "대체"한 것이 아니다. 각 도구가 가장 잘하는 영역에 배치하고, 불필요한 중간 단계를 제거한 것입니다.

### 4. 디자인 원칙은 도구에 종속되지 않는다

속도가 빨라져도 사용자 리서치, 문제 정의, 대안 탐색의 중요성은 줄지 않습니다. 오히려 실행 부담이 줄었기 때문에 사고에 더 집중할 수 있습니다.

---

## 마무리

> "디자이너로서 지금보다 더 흥미진진한 시기는 없었다."

Zych의 결론은 낙관적입니다. 새로운 도구와 작업 방식을 계속 시도하고, 프로세스를 진화시키겠다는 의지를 밝힌다.

Claude Code가 디자이너의 역할을 대체하지 않습니다. 오히려 **디자이너가 직접 통제할 수 있는 영역을 넓히고 있다**는 점이 핵심 메시지다. 의도를 해석하는 중간 계층이 줄어들수록, 디자이너의 비전이 결과물에 더 직접적으로 반영됩니다.
