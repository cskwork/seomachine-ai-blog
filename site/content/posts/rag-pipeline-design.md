## RAG란 무엇인가

RAG(Retrieval-Augmented Generation, 검색 증강 생성)는 LLM의 응답을 외부 지식으로 보강하는 아키텍처 패턴입니다.

LLM이 학습 데이터에만 의존하는 한계를 극복하기 위해, 실시간으로 관련 문서를 검색하여 컨텍스트로 제공합니다.

### RAG가 필요한 경우

- 최신 정보가 필요한 질의 (뉴스, 제품 업데이트)
- 조직 내부 문서 기반 질의응답
- 도메인 특화 지식이 필요한 작업
- 환각(Hallucination) 감소가 중요한 애플리케이션

## 파이프라인 구조

RAG 파이프라인은 크게 두 단계로 나뉩니다.

### 인덱싱 단계 (오프라인)

```
문서 수집 → 청킹 → 임베딩 → 벡터 저장소 적재
```

1. **문서 수집**: PDF, HTML, 마크다운 등 다양한 소스에서 문서 로드
2. **청킹(Chunking)**: 문서를 적절한 크기의 청크로 분할
3. **임베딩(Embedding)**: 각 청크를 벡터로 변환
4. **저장**: 벡터 데이터베이스에 인덱싱

### 검색-생성 단계 (온라인)

```
사용자 질의 → 질의 임베딩 → 유사도 검색 → 컨텍스트 구성 → LLM 생성
```

## 핵심 설계 결정

### 1. 청킹 전략

청크 크기와 방식은 검색 품질에 직접적인 영향을 미칩니다.

| 전략 | 설명 | 적합한 경우 |
|------|------|------------|
| 고정 크기 | 일정한 토큰/문자 수로 분할 | 균일한 문서 구조 |
| 의미 기반 | 문단, 섹션 단위로 분할 | 구조화된 문서 |
| 재귀적 | 큰 단위에서 작은 단위로 분할 | 다양한 문서 혼합 |
| 문장 윈도우 | 중심 문장 + 주변 문장 포함 | 세밀한 검색 필요 |

일반적으로 500-1000 토큰, 10-20% 오버랩이 출발점으로 권장됩니다.

### 2. 임베딩 모델 선택

- **OpenAI text-embedding-3-large**: 범용적, 높은 성능
- **Cohere embed-v3**: 다국어 지원이 우수
- **오픈소스 (BGE, E5)**: 비용 절감, 온프레미스 배포 가능

한국어 문서의 경우 다국어 모델의 성능을 반드시 벤치마크해야 합니다.

### 3. 벡터 데이터베이스

| 솔루션 | 특징 | 적합한 규모 |
|--------|------|------------|
| Pinecone | 관리형, 빠른 시작 | 중소 규모 |
| Weaviate | 하이브리드 검색, 자체 호스팅 | 중대 규모 |
| Qdrant | 고성능, 필터링 강점 | 대규모 |
| ChromaDB | 경량, 로컬 개발 | 프로토타입 |
| PostgreSQL + pgvector | 기존 인프라 활용 | 중소 규모 |

### 4. 검색 전략

단순 벡터 유사도 검색만으로는 한계가 있습니다.

- **하이브리드 검색**: 벡터 검색 + BM25 키워드 검색 결합
- **리랭킹(Reranking)**: 초기 검색 결과를 교차 인코더로 재정렬
- **질의 변환**: 사용자 질의를 검색에 최적화된 형태로 변환
- **메타데이터 필터링**: 날짜, 카테고리 등으로 검색 범위 제한

## 고급 패턴

### Agentic RAG

에이전트가 검색 전략을 동적으로 결정하는 패턴입니다.

- 질의 분석 후 적절한 검색 도구 선택
- 검색 결과가 불충분하면 질의를 재구성하여 재검색
- 여러 소스를 순차적으로 탐색

### Graph RAG

지식 그래프를 활용하여 엔티티 간 관계를 추론합니다.

- 문서에서 엔티티와 관계를 추출
- 그래프 데이터베이스에 저장
- 그래프 탐색으로 관련 정보를 수집
- 다중 홉(multi-hop) 추론이 가능

### Self-RAG

모델이 스스로 검색의 필요성을 판단하고, 검색 결과의 관련성을 평가합니다.

## 성능 최적화

### 평가 지표

| 지표 | 설명 | 목표 |
|------|------|------|
| Retrieval Precision | 검색된 문서 중 관련 문서 비율 | 80%+ |
| Retrieval Recall | 전체 관련 문서 중 검색된 비율 | 70%+ |
| Answer Faithfulness | 답변이 검색 결과에 기반하는 정도 | 90%+ |
| Answer Relevance | 답변이 질의에 관련된 정도 | 85%+ |

### 일반적인 문제와 해결

1. **검색 결과가 관련 없음**: 청킹 전략 재검토, 하이브리드 검색 도입
2. **환각이 여전히 발생**: 프롬프트에 "제공된 컨텍스트만 사용" 명시
3. **응답 속도가 느림**: 캐싱, 벡터 인덱스 최적화, 스트리밍 응답
4. **비용이 높음**: 청크 수 제한, 소형 임베딩 모델 사용

## 시작하기

RAG 파이프라인을 처음 구축한다면:

1. **작은 데이터셋으로 시작**: 100개 이내 문서로 프로토타입
2. **평가 세트 구축**: 질의-정답 쌍 20-50개 준비
3. **기본 파이프라인 구현**: 고정 크기 청킹 + 벡터 검색
4. **반복 개선**: 평가 결과를 기반으로 컴포넌트별 최적화
