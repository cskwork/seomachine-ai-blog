Claude Code를 처음 설치하고 사용하다 보면, 단순히 채팅창에 질문을 입력하는 것과 도구를 제대로 활용하는 것 사이에 큰 차이가 있다는 걸 곧 느끼게 됩니다. 핵심 명령어와 단축키를 숙지하는 것만으로도 작업 속도가 눈에 띄게 달라집니다.

이 글에서는 실제 개발 현장에서 매일 쓰는 명령어 20가지를 카테고리별로 정리했습니다. 단순 나열이 아니라, 언제 어떤 상황에서 쓰는지 실무 패턴과 함께 설명합니다.

## 대화 관리 명령어

### 1. `/help` - 도움말 표시

가장 기본적인 명령어지만 의외로 자주 잊히는 명령입니다. `/help`를 입력하면 현재 사용 가능한 모든 명령어 목록과 간단한 설명이 출력됩니다.

```
> /help
```

새 기능이 추가되거나 명령어가 바뀌었을 때 가장 먼저 확인해야 할 곳입니다. 특히 Claude Code 업데이트 직후에는 `/help`로 변경사항을 먼저 파악하는 습관을 들이면 좋습니다.

### 2. `/clear` - 대화 초기화

현재 세션의 대화 기록을 모두 지우고 새로 시작합니다. 이전 작업과 무관한 새로운 태스크를 시작할 때 사용합니다.

```
> /clear
```

Claude Code는 대화 기록 전체를 컨텍스트로 활용하기 때문에 중요합니다. 이전 작업의 파일명, 함수명, 설계 결정이 컨텍스트에 남아 있으면 새 작업에서 혼선이 생길 수 있습니다. 예를 들어, 오전에 Python 프로젝트를 작업하다가 오후에 Go 프로젝트로 전환할 때는 반드시 `/clear`로 컨텍스트를 정리하십시오.

### 3. `/compact` - 컨텍스트 압축

긴 세션에서 필수인 명령어입니다. 대화 기록을 압축하여 컨텍스트 창을 절약하면서도 핵심 정보는 유지합니다.

```
> /compact
```

컨텍스트 창이 80% 이상 채워지면 Claude의 응답 품질이 떨어질 수 있습니다. `/compact`는 지금까지의 대화를 요약하여 중요한 결정과 코드 상태는 보존하면서 불필요한 중간 과정을 제거합니다. 하루 종일 같은 프로젝트를 작업할 때, 점심 전후로 한 번씩 실행하는 것을 권장합니다.

### 4. `/config` - 설정 변경

테마, 사용 모델, 응답 언어 등 Claude Code의 동작 방식을 변경합니다.

```
> /config
```

대화형 메뉴가 열리며 현재 설정값을 확인하고 수정할 수 있습니다. 다크 모드 전환, 기본 모델을 Claude Opus에서 Sonnet으로 변경, 한국어 응답을 기본으로 설정하는 등의 작업을 여기서 처리합니다.

### 5. `/init` - CLAUDE.md 자동 생성

새 프로젝트에서 가장 먼저 실행해야 하는 명령어입니다. 현재 디렉토리의 코드를 분석하여 프로젝트에 맞는 `CLAUDE.md` 파일을 자동으로 생성합니다.

```
> /init
```

`CLAUDE.md`는 Claude Code가 매 대화 시작 시 자동으로 읽는 프로젝트 지침서입니다. 기술 스택, 코딩 컨벤션, 금지 명령어 등을 여기에 정의하면 매번 반복 설명할 필요가 없습니다. `/init`이 생성한 초안을 검토하고 팀 규칙에 맞게 수정하는 것이 프로젝트 셋업의 첫 단계입니다.

## 키보드 단축키

### 6. `Escape` - 현재 작업 중단

Claude가 응답을 생성하는 도중 멈추고 싶을 때 사용합니다. 잘못된 방향으로 가고 있다고 판단되면 즉시 `Escape`를 눌러 중단하십시오.

`Escape` 키는 완료된 응답을 취소하는 것이 아니라 현재 생성 중인 응답을 중단합니다. 이미 출력된 내용은 그대로 남으며, 그 시점까지의 결과를 활용할 수 있습니다. 특히 파일 수정 중 예상과 다른 방향으로 진행될 때 빠르게 제어권을 되찾는 핵심 수단입니다.

### 7. `Option+T` (macOS) / `Alt+T` - Extended Thinking 토글

Claude의 확장 사고 기능을 켜거나 끕니다. Extended Thinking이 활성화되면 Claude는 응답하기 전에 더 깊이 추론하는 과정을 거칩니다.

복잡한 아키텍처 결정, 버그의 근본 원인 분석, 알고리즘 설계처럼 신중한 판단이 필요한 작업에서는 Extended Thinking을 켜는 것이 좋습니다. 반면 단순한 코드 생성이나 파일 조작처럼 빠른 응답이 더 중요한 작업에서는 꺼두는 것이 효율적입니다.

### 8. `Ctrl+O` - Verbose 모드

Claude의 내부 사고 과정을 터미널에 출력합니다. Extended Thinking이 활성화된 상태에서 실제 추론 과정을 확인할 수 있습니다.

```
# Ctrl+O 활성화 후 Extended Thinking과 함께 사용
> 이 알고리즘의 시간 복잡도를 분석해줘

[thinking]
배열의 크기를 n이라 할 때, 외부 루프는 n번 실행되고...
[/thinking]

이 알고리즘의 시간 복잡도는 O(n²)입니다...
```

디버깅이 막혔을 때 Claude가 어떤 가정을 하고 있는지 파악하거나, Claude의 추론 품질을 검증하고 싶을 때 활용합니다.

### 9. `Ctrl+C` - 현재 생성 취소

`Escape`와 비슷하지만 동작이 다릅니다. `Ctrl+C`는 현재 생성을 즉시 취소하고 입력 프롬프트로 돌아갑니다. `Escape`가 "여기까지만 보여줘"라면 `Ctrl+C`는 "아예 취소해"에 해당합니다.

### 10. `Tab` - 자동 완성

파일 경로나 명령어를 입력할 때 `Tab` 키로 자동 완성을 사용할 수 있습니다. 긴 파일 경로를 입력할 때 오타를 줄이고 속도를 높이는 간단하지만 실용적인 기능입니다.

## 파일 작업 패턴

Claude Code는 자연어 명령을 인식하여 적절한 도구를 자동으로 선택합니다. 아래 패턴은 자주 쓰는 자연어 명령과 내부에서 활용되는 도구의 조합입니다.

### 11. 파일 읽기 - Read 도구 활용

```
> src/auth/middleware.ts 파일을 읽어줘
```

Claude는 Read 도구로 파일 전체 또는 특정 줄 범위를 읽습니다. 단순히 내용을 보여주는 데 그치지 않고 파일의 역할과 잠재적 문제점까지 분석해줍니다. 파일 경로를 정확히 명시하면 더 빠르게 처리됩니다.

### 12. 버그 분석 및 수정 제안

```
> 이 함수에서 null 참조 에러가 발생하고 있어. 원인을 찾고 수정해줘

function getUserName(user) {
  return user.profile.name.trim();
}
```

Claude는 코드를 분석하고, `user`, `user.profile`, `user.profile.name` 중 어느 단계에서 null이 될 수 있는지 순서대로 검토한 뒤 방어 코드를 제안합니다. 수정 사항을 바로 파일에 적용할지, 제안만 받을지 명시하면 더 정확한 결과를 얻습니다.

### 13. 패턴 검색 - Grep 도구 활용

```
> src/ 디렉토리에서 TODO 주석을 모두 찾아줘
```

```
> "deprecated" 키워드가 포함된 파일을 찾아줘
```

Claude는 Grep 도구로 정규식을 활용한 패턴 검색을 수행합니다. 대규모 코드베이스에서 특정 패턴, 함수 호출, 주석을 빠르게 찾아야 할 때 직접 터미널에서 `grep`을 실행하기보다 Claude에게 맡기면 결과 해석까지 한 번에 처리됩니다.

### 14. 파일 목록 조회 - Glob 도구 활용

```
> **/*.test.ts 패턴에 맞는 테스트 파일을 모두 찾아줘
```

```
> src/components/ 아래에 있는 모든 React 컴포넌트 파일을 나열해줘
```

Glob 패턴을 활용하면 특정 확장자, 경로 패턴에 해당하는 파일을 빠르게 조회할 수 있습니다. 리팩토링 전 영향 범위를 파악하거나 특정 레이어의 파일 목록을 확인할 때 유용합니다.

### 15. 파일 내용 수정 - Edit 도구 활용

```
> utils/format.ts에서 formatDate 함수의 날짜 포맷을 'YYYY-MM-DD'에서 'DD/MM/YYYY'로 변경해줘
```

Claude는 Edit 도구로 파일의 특정 부분만 정밀하게 수정합니다. 파일 전체를 덮어쓰지 않고 변경이 필요한 부분만 교체하므로 실수 위험이 낮습니다. 변경 전 어떤 내용을 어떻게 바꾸는지 확인한 뒤 승인하는 절차를 따르는 것을 권장합니다.

## 개발 워크플로우 패턴

### 16. TDD 워크플로우 - 테스트 작성 및 실행

```
> calculateTax 함수에 대한 단위 테스트를 먼저 작성하고, 그 다음 구현 코드를 만들어줘
```

Claude Code의 진가는 테스트 주도 개발에서 드러납니다. 테스트 먼저 작성 -> 실패 확인 -> 최소 구현 -> 통과 확인 순서로 요청하면 Claude가 각 단계를 차례로 실행하고 결과를 보고합니다. 이 패턴으로 테스트 없이 코드를 먼저 작성하는 나쁜 습관을 방지할 수 있습니다.

### 17. Git 통합 - 커밋 메시지 생성

```
> git diff를 보고 적절한 커밋 메시지를 만들어줘
```

```
> 스테이징된 변경사항을 분석하고 컨벤셔널 커밋 형식으로 커밋해줘
```

Claude는 `git diff`나 `git status` 결과를 분석하여 변경 의도를 파악하고 의미 있는 커밋 메시지를 작성합니다. `feat:`, `fix:`, `refactor:` 등 컨벤셔널 커밋 형식을 명시하면 일관된 커밋 이력을 유지할 수 있습니다.

### 18. 에러 로그 디버깅

```
> 다음 에러 로그를 분석하고 원인과 해결책을 알려줘

TypeError: Cannot read properties of undefined (reading 'map')
    at ProductList.render (ProductList.jsx:42:23)
    at processChild (/node_modules/react-dom/...)
```

에러 메시지, 스택 트레이스, 재현 조건을 함께 제공할수록 진단이 더 정확해집니다. Claude는 에러 타입, 발생 위치, 가능한 원인을 순서대로 분석하고 수정 방법을 제시합니다.

### 19. 빌드 및 린트 실행

```
> npm run build를 실행하고 에러가 있으면 수정해줘
```

```
> eslint를 실행하고 자동 수정 가능한 이슈는 바로 고쳐줘
```

Claude Code는 Bash 도구로 빌드 명령어를 직접 실행할 수 있습니다. 빌드 실패 시 에러 메시지를 자동으로 파악하고 수정 방안을 제시하거나, 허용된 경우 직접 수정까지 수행합니다. 코드를 작성한 직후 빌드와 린트를 자동 실행하도록 요청하는 것이 좋습니다.

### 20. PR 코드 리뷰

```
> 이 PR의 변경사항을 리뷰해줘. 버그, 성능 이슈, 코드 스타일 문제를 중점으로 봐줘
```

```
> git diff main...HEAD를 기준으로 보안 취약점이 있는지 확인해줘
```

코드 리뷰를 요청할 때는 어떤 관점에서 검토할지 명시하면 더 집중된 피드백을 받을 수 있습니다. 성능, 보안, 가독성, 테스트 커버리지 중 우선순위를 함께 전달하십시오.

## 실무 인사이트: 자주 쓰는 명령어 조합

단일 명령어보다 여러 패턴을 조합할 때 진정한 생산성 향상이 나타납니다.

### 디버깅 조합

에러가 발생했을 때 다음 순서로 진행합니다.

```
# 1단계: 에러 분석
> 다음 에러 로그를 분석해줘: [에러 붙여넣기]

# 2단계: 관련 코드 조회
> 에러가 발생한 파일을 읽고 문제 부분을 찾아줘

# 3단계: 수정 적용
> 원인을 수정하고 테스트를 실행해서 해결됐는지 확인해줘
```

이 조합의 핵심은 "분석 -> 위치 파악 -> 수정 -> 검증"을 하나의 흐름으로 처리하는 것입니다. 중간에 맥락이 끊기면 처음부터 다시 설명해야 하므로, `/clear` 없이 같은 세션 안에서 완결짓는 것이 효율적입니다.

### 리팩토링 조합

```
# 1단계: 패턴 파악
> 코드베이스에서 any 타입을 사용하는 파일을 모두 찾아줘

# 2단계: 우선순위 결정
> 찾은 파일들 중 수정이 가장 간단한 것부터 우선순위를 매겨줘

# 3단계: 순차 수정
> users.service.ts부터 any를 적절한 타입으로 교체해줘

# 4단계: 검증
> 변경 후 타입 에러가 없는지 확인하고 테스트를 실행해줘
```

### `/compact`를 언제 사용해야 하는가

`/compact` 사용 기준은 컨텍스트 사용량이 아니라 대화의 "신선도"입니다.

사용해야 하는 시점:
- 하나의 작업이 완료되고 다음 작업으로 넘어갈 때
- 긴 시행착오 끝에 해결책을 찾았을 때 (중간 과정은 불필요)
- 응답 속도가 눈에 띄게 느려졌을 때

사용하지 말아야 하는 시점:
- 아직 해결 중인 복잡한 버그를 디버깅할 때 (중간 추론 과정이 필요)
- 코드 리뷰 세션 중 (변경의 맥락이 전부 중요)
- 연속적인 리팩토링 작업 중 (이전 수정이 다음 수정의 기준이 됨)

`/compact`는 맥락을 잃지 않으면서 컨텍스트 창을 확보하는 도구이지, 단순히 메모리를 절약하는 도구가 아닙니다. 타이밍을 잘못 잡으면 Claude가 이전 결정의 이유를 잊어버려 같은 실수를 반복할 수 있습니다.

명령어를 하나하나 외우려 하기보다, 오늘 소개한 조합 패턴부터 실제 작업에 적용해 보십시오. 디버깅할 때 에러 로그를 그냥 붙여넣는 것과 단계별로 조합하는 것의 차이를 직접 경험하면 자연스럽게 습관이 됩니다.
