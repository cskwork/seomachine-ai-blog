Claude Code를 처음 사용하면 놀랄 만큼 긴 대화가 가능하다는 사실에 감탄하게 됩니다. 수십 번의 주고받기를 거치며 복잡한 리팩토링을 완료하고, 디버깅 세션을 마무리하고, 새 기능을 구현하는 동안 Claude는 앞선 대화를 모두 기억하는 것처럼 보입니다.

하지만 어느 순간 이상한 일이 벌어지기 시작합니다. 분명히 이전에 설명했던 아키텍처 결정을 Claude가 다시 물어봅니다. 아까 읽었던 파일을 다시 읽으려 합니다. 이미 합의한 코딩 컨벤션과 다른 스타일의 코드를 작성합니다.

이것은 Claude의 능력 한계가 아닙니다. 컨텍스트 윈도우가 한계에 가까워졌다는 신호입니다.

## 컨텍스트 윈도우란

컨텍스트 윈도우란 한 대화 세션에서 Claude가 동시에 참조할 수 있는 토큰의 총량을 의미합니다. 토큰은 텍스트를 분해하는 단위로, 한국어의 경우 대략 글자 2-3개가 하나의 토큰에 해당합니다.

여기서 중요한 점은 컨텍스트를 소모하는 주체가 생각보다 다양하다는 것입니다. 사용자가 입력하는 메시지뿐 아니라, Claude가 도구를 호출한 결과(파일 읽기, 검색 결과, 빌드 출력), 그리고 Claude 자신이 작성하는 응답 텍스트 모두 컨텍스트에 누적됩니다. 즉, 대화가 길어질수록, 파일을 많이 읽을수록, 검색을 자주 할수록 컨텍스트는 빠르게 차오릅니다.

한도에 도달하면 어떻게 될까요? Claude Code는 이전 대화를 자동으로 압축합니다. 이 자동 압축이 문제의 핵심입니다. 압축은 Claude가 판단하기에 덜 중요하다고 여기는 내용을 요약하거나 제거하는 방식으로 작동하는데, 이 과정에서 여러분이 중요하게 여기는 맥락이 사라질 수 있습니다. 오류 메시지의 세부 내용, 특정 설계 결정의 근거, 이미 시도하고 실패한 방법들. 이런 정보가 증발하면서 Claude는 마치 처음 만난 사람처럼 행동하기 시작합니다.

## 컨텍스트를 많이 소모하는 패턴

컨텍스트 관리의 첫 번째 단계는 무엇이 토큰을 빠르게 소모하는지 파악하는 것입니다.

### 대용량 파일 전체 읽기

1,000줄짜리 파일을 통째로 읽어달라고 요청하면 그 1,000줄이 고스란히 컨텍스트를 차지합니다. 실제로 필요한 내용이 특정 섹션에 집중되어 있을 때도 마찬가지입니다. 해결책은 범위를 지정하는 것입니다. "100번째 줄부터 150번째 줄까지만 읽어줘"처럼 구체적으로 요청하면 컨텍스트 소모를 대폭 줄일 수 있습니다. 파일의 전반적인 구조가 필요하다면 전체를 읽는 대신 "이 파일에 어떤 함수들이 있는지 목록만 보여줘"처럼 요약을 먼저 요청하는 방법도 있습니다.

### 도구 호출 결과 누적

Grep 검색이나 Glob 파일 탐색의 결과가 수백 줄이면 그 결과 전체가 컨텍스트에 쌓입니다. "전체 프로젝트에서 `import` 키워드 검색해줘"라는 요청은 수백, 수천 개의 결과를 불러올 수 있습니다. 검색 범위를 좁히거나, 결과 수를 제한하는 것이 효과적입니다. "src/ 디렉토리 안에서만 검색해줘"처럼 경로를 제한하거나, 도구에서 지원한다면 결과 수 제한 옵션을 활용하세요.

### 반복적인 빌드와 테스트 출력

개발 과정에서 빌드나 테스트를 여러 번 반복하면 매번 수십에서 수백 줄의 출력이 컨텍스트에 추가됩니다. 특히 테스트가 많은 프로젝트에서는 전체 테스트 실행 결과가 상당한 토큰을 차지합니다. "실패한 테스트만 보여줘"라는 요청 하나로 불필요한 컨텍스트 소모를 크게 줄일 수 있습니다. 성공 케이스의 출력은 대부분 그 순간에만 필요하지, 이후 대화에서 다시 참조할 일이 없습니다.

### 불필요하게 넓은 탐색 범위

"이 프로젝트 전체를 분석해줘"보다 "이 파일의 이 함수가 어디서 호출되는지 찾아줘"처럼 구체적인 요청이 컨텍스트를 아낍니다. 탐색 범위를 좁힐수록 Claude가 처리하는 데이터양이 줄고, 그만큼 컨텍스트를 절약할 수 있습니다.

## /compact 완벽 가이드

`/compact` 명령은 Claude Code가 제공하는 컨텍스트 관리 도구입니다. 이 명령을 이해하고 적절한 시점에 활용하면 긴 세션을 훨씬 효율적으로 운영할 수 있습니다.

### 언제 사용하는가

작업의 주제가 바뀔 때가 가장 적절한 시점입니다. 예를 들어 특정 버그를 디버깅하는 긴 세션을 마치고 새로운 기능을 구현하기 시작할 때, 이전 디버깅 과정의 세부 내용은 더 이상 필요하지 않습니다. 이때 `/compact`를 실행하면 디버깅 과정의 상세 내용은 압축하고 최종 해결책만 요약으로 남겨 컨텍스트를 확보할 수 있습니다.

대화 횟수가 30-40회를 넘어가기 시작했다면 선제적으로 `/compact`를 고려해볼 시점입니다. 자동 압축이 예측 불가능한 방식으로 일어나도록 내버려두는 것보다, 적절한 시점에 능동적으로 압축을 실행하는 편이 중요한 맥락을 보존하는 데 유리합니다.

Claude가 이전에 명확하게 지시한 사항을 따르지 않거나 이전 결정사항을 기억하지 못하는 것처럼 보일 때도 `/compact`가 도움이 됩니다. 이미 압축이 일어나 맥락이 희박해진 상태에서 계속 진행하는 것보다, 의도적으로 요약을 정리하고 재시작하는 편이 낫습니다.

### 어떻게 작동하는가

`/compact`는 현재까지의 대화 전체를 분석하여 핵심 정보만 추출한 요약으로 대체합니다. CLAUDE.md에 저장된 프로젝트 설정은 이 과정에서 영향을 받지 않고 유지됩니다. 진행 중이던 작업의 현재 상태, 완료된 단계, 남은 작업 등은 요약에 포함되어 이어서 작업하는 데 지장이 없습니다.

### 주의해야 할 점

압축은 본질적으로 정보 손실을 수반합니다. 코드 변경 내용의 세부 사항, 특정 결정을 내린 근거, 검토했다가 기각한 대안 등이 요약 과정에서 사라질 수 있습니다. 따라서 `/compact`를 실행하기 전에 이후에도 중요하게 참조될 결정사항이나 컨벤션을 CLAUDE.md에 기록해두는 습관이 중요합니다. 압축 전의 기록이 압축 후 세션의 품질을 결정합니다.

## 세션 분리 전략

`/compact`가 하나의 세션 안에서 컨텍스트를 관리하는 전략이라면, 세션 자체를 분리하는 것도 유효한 접근입니다.

### 새 세션이 필요한 시점

완전히 다른 성격의 작업을 시작할 때입니다. 예를 들어 A 기능의 버그를 수정하는 세션을 마치고, 이제 B 기능을 새로 만들어야 할 때입니다. 이 두 작업 사이에 공유되는 맥락이 거의 없다면 새 세션이 더 깔끔합니다.

대규모 리팩토링을 완료하고 그 결과물 위에서 새 기능을 구현하는 경우도 마찬가지입니다. 리팩토링 과정의 상세한 변경 내역은 새 기능 구현 단계에서는 불필요한 노이즈가 됩니다. 리팩토링을 커밋으로 마무리하고 새 세션에서 신선하게 시작하는 편이 낫습니다.

### 세션 간 맥락을 효과적으로 전달하는 방법

세션을 나누면 맥락이 끊어지는 것이 당연한 걱정입니다. 하지만 올바른 습관 몇 가지로 이 문제를 충분히 해결할 수 있습니다.

CLAUDE.md를 살아있는 문서로 관리하세요. 단순히 초기 프로젝트 설정만 기록하는 것이 아니라, 세션을 거치며 결정된 아키텍처 선택, 팀 컨벤션, 진행 중인 작업의 현재 상태를 계속 업데이트합니다. 이렇게 하면 새 세션을 시작할 때 Claude가 CLAUDE.md를 읽는 것만으로도 빠르게 맥락을 파악할 수 있습니다.

새 세션의 첫 메시지를 명확하게 구성하는 것도 중요합니다. "이전 세션에서 사용자 인증 모듈 리팩토링을 완료했고, JWT 토큰 기반으로 변경했습니다. 이제 이 위에서 소셜 로그인 기능을 추가해야 합니다."처럼 이전 세션의 결론과 현재 시작점을 명확히 전달하면 연속성이 유지됩니다.

커밋 메시지를 상세하게 작성하는 습관도 도움이 됩니다. `git log`가 하나의 문서처럼 읽힐 수 있도록 각 커밋에 무엇을 왜 변경했는지 기록해두면, 새 세션에서 코드베이스의 진화 과정을 빠르게 파악할 수 있습니다.

## 실무 인사이트: 컨텍스트 부족의 3가지 징후

컨텍스트 관리의 핵심은 문제가 심각해지기 전에 징후를 알아채는 것입니다. 실제 사용 경험에서 반복적으로 나타나는 세 가지 징후를 소개합니다.

**징후 1: 같은 파일을 반복해서 읽는다.** 이미 이 세션에서 읽었던 파일을 Claude가 다시 읽으려 한다면, 이전에 읽은 내용이 컨텍스트에서 밀려났거나 압축된 것입니다. 이 시점에서 `/compact`를 실행하거나, CLAUDE.md에 해당 파일의 핵심 내용을 요약해두는 것이 좋습니다.

**징후 2: 이미 합의한 결정사항을 무시하거나 다시 질문한다.** 초반 대화에서 명확하게 합의한 아키텍처 방향, 사용할 라이브러리, 코딩 패턴 등을 Claude가 다시 제안하거나 다른 방향으로 구현하기 시작한다면 그 합의 내용이 압축 과정에서 사라진 것입니다. 즉각적인 대처는 그 결정사항을 다시 명시적으로 알려주는 것이지만, 근본적인 해결은 중요한 결정사항을 CLAUDE.md에 기록하는 것입니다.

**징후 3: 코드 스타일이 갑자기 달라진다.** 세션 초반에 확립된 코딩 스타일이나 네이밍 컨벤션과 다른 코드를 생성하기 시작한다면, CLAUDE.md에 명시되지 않은 암묵적인 컨벤션 정보가 사라진 신호입니다. 이 경우 세션 초반에 나눴던 스타일 관련 논의를 CLAUDE.md에 공식적으로 기록하고, 필요하다면 `/compact`를 통해 정리된 상태에서 재시작하는 것을 권장합니다.

이 세 가지 징후 중 하나라도 발견한다면, 즉시 멈추고 컨텍스트 상태를 점검하는 것이 시간을 아끼는 길입니다. 컨텍스트가 부족한 상태에서 계속 진행하면 잘못된 방향으로 작업이 쌓여나가고, 나중에 더 많은 시간을 들여 수정해야 하는 상황이 됩니다.

## 체계적인 컨텍스트 관리 루틴

앞서 소개한 내용들을 종합하여 실용적인 루틴을 제안합니다.

세션을 시작할 때는 CLAUDE.md를 최신 상태로 유지하고, 이번 세션에서 달성할 목표를 명확하게 첫 메시지에 담습니다. 세션 진행 중에는 파일 읽기와 검색의 범위를 가능한 한 좁게 유지하고, 작업 단위를 완료할 때마다 커밋하여 진행 상황을 git에 기록합니다. 세션을 마칠 때는 다음 세션에 필요한 맥락을 CLAUDE.md에 업데이트하고, 필요하다면 `/compact`로 정리한 뒤 세션을 종료합니다.

이 루틴이 처음에는 번거롭게 느껴질 수 있습니다. 하지만 한 번 컨텍스트 오염으로 인해 세션 전체를 다시 시작하는 경험을 해보면, 예방적 관리가 얼마나 효율적인지 직감적으로 이해하게 됩니다.

컨텍스트 윈도우는 제한된 자원입니다. 이 자원을 의식적으로 관리하는 것이 Claude Code를 전문적으로 활용하는 핵심 역량 중 하나입니다.
