Claude Code를 단일 에이전트로 사용하는 것만으로도 생산성이 크게 올라가지만, 진짜 위력은 멀티 에이전트 오케스트레이션에 있다. 대규모 리팩토링, 동시다발적 파일 수정, 여러 관점의 코드 리뷰처럼 본질적으로 병렬화할 수 있는 작업에서 멀티 에이전트는 단순 속도 향상을 넘어 작업 방식 자체를 바꾼다.

이 글에서는 서브에이전트의 작동 원리부터 실전 병렬 실행 패턴, git worktree를 이용한 격리 기법, 그리고 대규모 리팩토링 예제까지 단계적으로 다룬다.

## 서브에이전트란

서브에이전트는 메인 에이전트가 `Task` 도구를 통해 독립적 하위 작업을 위임하는 대상이다. 핵심 특성은 세 가지다.

첫째, **별도 컨텍스트에서 실행**된다. 서브에이전트는 메인 에이전트의 컨텍스트 창을 공유하지 않는다. 서브에이전트가 수십 개 파일을 탐색하고 분석하더라도, 그 과정에서 소비한 토큰은 메인 에이전트에 영향을 주지 않는다.

둘째, **결과만 반환**한다. 서브에이전트는 작업을 완료한 후 요약된 결과만 메인 에이전트에게 돌려준다. 메인 에이전트 입장에서는 상세한 탐색 과정을 볼 필요 없이 최종 결론만 받아 처리하면 된다.

셋째, **독립적으로 도구를 사용**한다. 서브에이전트는 파일 읽기, 코드 수정, 명령어 실행 등 허용된 도구 전체를 독립적으로 사용할 수 있다.

팀 리더가 팀원들에게 작업을 분배하는 방식에 비유할 수 있다. 리더는 각 팀원에게 명확한 목표와 책임 범위를 정해주고, 결과 보고를 받아 전체 작업을 조율한다. 리더가 팀원의 모든 중간 과정에 개입하거나 모니터링할 필요는 없다.

## 서브에이전트 유형

### 내장 에이전트

Claude Code는 별도 설정 없이 사용할 수 있는 내장 에이전트 유형을 제공한다.

**general-purpose**는 가장 범용적인 에이전트로, 코드 검색, 다단계 작업, 일반적인 조사 업무에 적합하다. 특별한 전문성보다 폭넓은 유연성이 필요한 작업에 사용한다.

**Explore**는 코드베이스를 빠르게 탐색하는 데 특화된 에이전트다. 파일 패턴 검색, 키워드 검색, 디렉토리 구조 파악처럼 코드를 수정하지 않고 이해하는 데 집중하는 작업에 적합하다.

**Plan**은 구현 전략 설계와 아키텍처 분석을 담당한다. 코드를 작성하기 전에 접근 방식을 결정하거나, 기존 코드베이스를 분석해 최적의 변경 지점을 찾는 데 사용한다.

### 커스텀 에이전트

프로젝트 고유의 워크플로우와 컨벤션이 있다면 커스텀 에이전트를 정의하는 편이 훨씬 효과적이다. `.claude/agents/` 디렉토리에 마크다운 파일로 작성하면 된다.

```
.claude/
└── agents/
    ├── code-reviewer.md
    ├── security-reviewer.md
    ├── tdd-guide.md
    └── refactor-cleaner.md
```

각 파일에 에이전트의 역할, 작업 방식, 사용 가능한 도구, 출력 형식 등을 명시한다. 예를 들어 `security-reviewer.md`에는 OWASP Top 10 기준으로 코드를 검토하고, 취약점 심각도를 CRITICAL/HIGH/MEDIUM/LOW로 분류해 반환하라는 지침을 담는다.

커스텀 에이전트의 장점은 일관성이다. 메인 에이전트에게 매번 "보안 관점에서 리뷰해줘"라고 설명하는 대신, `security-reviewer`를 호출하면 언제나 동일한 기준과 형식으로 결과를 받을 수 있다.

## 병렬 실행 패턴

### 패턴 1: 독립 파일 동시 수정

가장 직관적인 병렬화 패턴이다. 서로 의존 관계가 없는 파일을 여러 에이전트가 동시에 처리한다.

```
오케스트레이터
├── 에이전트 A: src/components/Button.tsx 수정
├── 에이전트 B: src/components/Input.tsx 수정
├── 에이전트 C: src/components/Modal.tsx 수정
└── 에이전트 D: src/components/Table.tsx 수정
```

예를 들어 10개 React 컴포넌트를 JavaScript에서 TypeScript로 마이그레이션한다면, 10개를 순차 처리하는 대신 5개씩 두 배치로 나눠 병렬 실행하면 이론상 절반의 시간으로 줄어든다. 전제 조건은 파일 간 의존성이 없어야 한다는 것이다. Button.tsx를 수정하는 에이전트와 Input.tsx를 수정하는 에이전트가 서로의 작업을 참조하거나 영향을 주지 않아야 한다.

### 패턴 2: 리서치와 구현 분리

대규모 변경 작업에 효과적인 패턴이다. 코드베이스를 탐색하고 패턴을 분석하는 리서치 단계와 실제 구현 단계를 분리한다.

순서는 순차적이지만, 리서치 에이전트가 메인 에이전트의 컨텍스트를 소모하지 않는다는 점이 핵심이다. 리서치 에이전트는 수백 개의 파일을 뒤지고 패턴을 분석하면서 자신의 컨텍스트를 사용한다. 메인 에이전트는 최종 분석 보고서만 받아 구현 에이전트에게 전달한다.

```
1단계: 리서치 에이전트
   - 전체 코드베이스에서 에러 핸들링 패턴 탐색
   - 중복된 패턴, 불일치한 방식, 개선 포인트 정리
   - 결과: 2페이지 분석 보고서 반환

2단계: 구현 에이전트 (분석 보고서 기반)
   - 통일된 에러 핸들링 패턴 적용
```

### 패턴 3: 멀티 관점 리뷰

같은 코드 변경을 여러 전문가 관점으로 동시에 검토하는 패턴이다. 하나의 코드 diff를 세 에이전트에게 동시에 전달한다.

```
코드 변경 diff
├── 에이전트 A (security-reviewer): 인증/인가, 입력 검증, 취약점 분석
├── 에이전트 B (performance-reviewer): N+1 쿼리, 불필요한 연산, 캐싱 기회
└── 에이전트 C (convention-reviewer): 네이밍, 파일 구조, 코딩 스타일
```

세 에이전트가 병렬로 실행되고, 결과가 모두 모이면 오케스트레이터가 종합해 최종 리뷰 리포트를 생성한다. 순차적으로 세 번 리뷰하는 것보다 시간이 크게 줄어들고, 각 에이전트가 자신의 전문 영역에 집중하므로 품질도 높다.

## git worktree를 활용한 격리

병렬 실행 패턴 1처럼 여러 에이전트가 동시에 파일을 수정할 때, 같은 디렉토리에서 작업하면 충돌이 발생할 수 있다. 이 문제를 해결하는 것이 git worktree 격리다.

`isolation: "worktree"` 옵션을 쓰면 각 에이전트에게 격리된 저장소 사본이 제공된다. 에이전트는 독립된 워킹 디렉토리에서 자유롭게 파일을 수정하고, 작업 완료 후 변경사항을 메인 브랜치에 병합한다. 변경사항이 없거나 작업이 취소되면 worktree는 자동으로 정리된다.

```
main session (main branch)
├── agent A → worktree: feature-auth
│   └── /tmp/worktree-a/ (독립된 레포 사본)
├── agent B → worktree: feature-api
│   └── /tmp/worktree-b/ (독립된 레포 사본)
└── agent C → worktree: feature-tests
    └── /tmp/worktree-c/ (독립된 레포 사본)
```

각 에이전트는 자신의 worktree에서만 작업한다. 에이전트 A가 `src/auth/login.ts`를 수정하는 동안 에이전트 B가 같은 파일의 다른 버전을 수정해도 충돌이 없다. 모든 에이전트 작업이 완료되면 오케스트레이터가 각 worktree의 변경사항을 순서에 맞게 메인 브랜치에 병합한다.

worktree 격리를 쓸 때는 작업 범위를 명확히 분리해야 한다. 에이전트 A는 `src/auth/` 디렉토리만, 에이전트 B는 `src/api/` 디렉토리만 담당하도록 경계를 명시해야 병합 시 충돌을 최소화할 수 있다.

## 실전 예제: 대규모 리팩토링을 3개 에이전트로 분할

시나리오: Express 앱 전체의 에러 핸들링을 제각각인 기존 방식에서 통일된 커스텀 에러 클래스 체계로 리팩토링한다.

**1단계: 오케스트레이터 분석**

오케스트레이터가 먼저 전체 코드베이스의 에러 핸들링 패턴을 탐색한다. `routes/`, `services/`, `middleware/` 세 영역을 독립적으로 리팩토링할 수 있다고 판단하면 작업을 분할한다.

```
분석 결과:
- routes/: try-catch가 없거나 일관성 없는 res.status() 사용 (12개 파일)
- services/: 에러를 문자열로 throw하거나 console.error로 묻어버림 (8개 파일)
- middleware/: 글로벌 에러 핸들러 없음, 에러가 Express 기본 핸들러로 전달됨
```

**2단계: 3개 에이전트 병렬 실행**

```
에이전트 A (worktree: refactor-routes)
  담당: routes/ 디렉토리 12개 파일
  목표: 모든 route handler에 try-catch 추가,
        에러 발생 시 next(error) 호출로 통일

에이전트 B (worktree: refactor-services)
  담당: services/ 디렉토리 8개 파일
  목표: 커스텀 에러 클래스(AppError, ValidationError, NotFoundError) 적용,
        문자열 throw를 에러 클래스 인스턴스로 교체

에이전트 C (worktree: refactor-middleware)
  담당: middleware/ 디렉토리
  목표: 글로벌 에러 핸들러 미들웨어 구현,
        에러 타입별 HTTP 상태 코드 매핑,
        개발/운영 환경별 에러 응답 형식 분리
```

**3단계: 오케스트레이터 통합**

세 에이전트의 작업이 끝나면 오케스트레이터가 각 worktree를 순서대로 병합한다. 먼저 에이전트 C의 커스텀 에러 클래스 정의를 병합한 뒤, 에이전트 A와 B의 변경사항을 병합한다. 의존 관계가 있으면 병합 순서를 고려해야 한다.

마지막으로 오케스트레이터가 통합 테스트를 실행해 전체 파이프라인이 정상 동작하는지 확인한다.

이 작업을 단일 에이전트로 순차 처리했다면 파일 하나씩 수정하면서 전체 컨텍스트를 소모했을 것이다. 멀티 에이전트 방식은 각 에이전트가 자기 영역에만 집중하므로 컨텍스트 효율이 높고, 병렬 실행으로 전체 소요 시간도 줄어든다.

## 실무 인사이트: 멀티 에이전트의 함정

### 함정 1: 같은 파일 동시 수정

여러 에이전트가 같은 파일을 동시에 수정하면 나중에 끝난 에이전트의 변경사항이 이전 에이전트의 작업을 덮어쓰거나 충돌이 발생한다. worktree 격리를 쓰면 작업 중 충돌은 방지되지만, 병합 시 충돌이 발생할 수 있다.

해결책은 두 가지다. 첫째, 작업 범위를 파일 단위로 명확히 분리해 에이전트 간 파일 중복이 없게 한다. 둘째, 같은 파일을 반드시 여러 에이전트가 다뤄야 한다면 순차 처리로 전환한다.

### 함정 2: 오케스트레이터 컨텍스트 과부하

서브에이전트가 컨텍스트를 보호해준다고 했지만, 에이전트 결과를 무분별하게 수집하면 오케스트레이터의 컨텍스트가 금방 찬다. 에이전트 5개의 결과를 각각 2,000토큰씩 받으면 오케스트레이터 컨텍스트에 10,000토큰이 쌓인다.

에이전트를 호출할 때 출력 형식을 명시적으로 제한해야 한다. "분석 결과를 5줄 이내로 요약해줘", "변경된 파일 목록과 변경 이유만 반환해줘"처럼 구체적으로 출력 형식을 지시한다.

### 함정 3: 과도한 병렬화

모든 작업을 병렬로 실행하면 오히려 효율이 떨어지는 경우가 있다. 에이전트 생성과 관리, 결과 수집과 통합에도 비용이 든다. 간단한 작업을 3개 에이전트에게 분배하면 조율 비용이 실제 작업 시간보다 클 수 있다.

병렬화 판단 기준은 간단하다. 독립적으로 실행 가능한 작업이 3개 이상이고, 각 작업을 단독 실행했을 때 최소 몇 분 이상 걸릴 것으로 예상될 때 병렬화를 적용한다. 간단한 파일 수정 몇 개는 메인 에이전트가 순차적으로 처리하는 것이 더 빠르다.

## 멀티 에이전트 설계 원칙

멀티 에이전트 시스템을 설계할 때 지켜야 할 원칙을 정리하면 다음과 같다.

**명확한 경계 정의**: 각 에이전트의 담당 범위를 파일, 디렉토리, 기능 단위로 명확하게 정의한다. 경계가 모호하면 중복 작업과 충돌이 발생한다.

**출력 형식 표준화**: 모든 에이전트가 오케스트레이터가 처리하기 쉬운 형식으로 결과를 반환하게 지시한다. 자유 형식 텍스트보다 구조화된 형식이 통합하기 쉽다.

**의존성 그래프 파악**: 에이전트 간 작업 의존성을 미리 파악한다. A가 완료되어야 B가 시작할 수 있다면 순차 처리해야 한다. 의존성이 없는 작업만 병렬화한다.

**점진적 복잡도 증가**: 처음에는 2개 에이전트로 시작해 패턴을 익힌 후 점차 규모를 늘린다. 한 번에 10개 에이전트를 오케스트레이션하려 하면 디버깅이 매우 어려워진다.

멀티 에이전트 오케스트레이션은 강력하지만 그만큼 복잡하다. 올바른 패턴을 선택하고 함정을 피하면, 혼자서는 시간이 너무 오래 걸려 엄두를 못 냈던 대규모 작업을 현실적인 시간 안에 처리할 수 있다.
