## Claude Code란 무엇인가

Claude Code는 Anthropic이 2025년 공개한 CLI 기반 AI 코딩 도구다. 이름에 "Code"가 붙어 있어 코드 편집기를 떠올리기 쉽지만, 실제로는 터미널에서 직접 실행하는 명령줄 에이전트다. 별도의 IDE 플러그인이나 GUI 없이, 이미 쓰고 있는 터미널 환경에서 대화 형식으로 코딩 작업을 처리한다.

동작 방식이 기존 AI 코딩 도구와 근본적으로 다르다. GitHub Copilot이나 Cursor는 편집기 안에 AI 기능을 내장하는 방식이다. 코드를 타이핑하면 자동 완성 제안이 나타나거나, 선택한 코드 블록의 설명을 요청하는 식이다. 이 도구들은 편집기가 중심이고 AI는 보조다.

Claude Code는 반대다. AI 에이전트가 중심에 있고, 파일 시스템과 터미널이 그 에이전트의 작업 공간이다. Claude Code는 프로젝트 파일을 직접 읽고, 수정하고, 셸 명령을 실행한다. 작업 하나를 요청하면 에이전트가 관련 파일을 탐색하고, 필요한 변경을 계획하고, 실행까지 마친 뒤 결과를 보고한다. 단순한 코드 제안이 아니라, 작업을 완수하는 방식이다.

이런 에이전트 방식 덕분에 Claude Code는 특히 다음 상황에서 강점을 발휘한다. 처음 보는 레거시 코드베이스를 빠르게 파악해야 할 때, 여러 파일에 걸친 리팩터링을 할 때, 반복적인 보일러플레이트 생성을 자동화할 때다. 반면 실시간 자동 완성이 필요한 빠른 코딩에는 Copilot 계열 도구가 더 적합하다. 두 도구는 경쟁 관계가 아니라 용도가 다르다.

## 설치 방법

### 사전 요건: Node.js 18 이상

Claude Code는 Node.js 기반으로 배포된다. 설치 전에 Node.js 버전을 확인한다.

```bash
node --version
```

`v18.0.0` 이상이 출력되면 준비된 것이다. 그보다 낮은 버전이거나 Node.js가 없다면 [nodejs.org](https://nodejs.org)에서 LTS 버전을 설치한다. `nvm`을 사용하는 개발자는 다음과 같이 버전을 맞춘다.

```bash
nvm install --lts
nvm use --lts
```

### Claude Code 설치

npm으로 전역 설치한다.

```bash
npm install -g @anthropic-ai/claude-code
```

설치가 끝나면 버전을 확인해 제대로 설치됐는지 검증한다.

```bash
claude --version
```

정상 설치라면 `claude/1.x.x` 형태의 버전 문자열이 출력된다.

### 첫 실행과 인증

처음 `claude` 명령을 실행하면 인증 단계가 시작된다.

```bash
claude
```

두 가지 인증 방법 중 하나를 선택할 수 있다. 첫 번째는 Anthropic Max 플랜 구독을 통한 인증이다. 브라우저가 열리면서 Anthropic 계정에 로그인하라는 안내가 나온다. 로그인을 완료하면 터미널로 돌아와 바로 사용할 수 있다. 두 번째는 API 키를 직접 입력하는 방식이다. Anthropic Console에서 발급받은 키를 아래 환경 변수로 설정하면 된다.

```bash
export ANTHROPIC_API_KEY="sk-ant-..."
```

API 키 방식은 사용량에 따라 과금되므로, 사용 전에 요금 정책을 미리 확인하는 것이 좋다.

인증이 완료되면 다음과 같은 프롬프트가 나타난다.

```
Claude Code  (type /help for help)

>
```

이제 대화를 시작할 수 있다.

## 첫 번째 작업: 실습으로 배우기

설치 직후 바로 "Hello World"급 실습을 해보는 것이 가장 빠르게 익히는 방법이다. 아래 다섯 단계를 순서대로 따라 하면 Claude Code의 핵심 동작 방식을 체감할 수 있다.

### Step 1: 프로젝트 디렉토리에서 실행

Claude Code는 실행 위치의 파일 시스템을 작업 범위로 삼는다. 따라서 반드시 작업할 프로젝트의 루트 디렉토리에서 실행해야 한다.

```bash
cd ~/my-project
claude
```

홈 디렉토리나 임의 위치에서 실행하면 Claude가 프로젝트 구조를 파악하지 못해 엉뚱한 경로에 파일을 찾거나 생성할 수 있다. 이 점은 뒤에서 흔한 실수로 다시 다룬다.

### Step 2: 프로젝트 구조 탐색

첫 메시지로 프로젝트 개요를 물어보면 좋은 출발점이 된다.

```
이 프로젝트의 구조를 설명해줘
```

Claude가 디렉토리 구조를 탐색하고, 주요 파일과 설정의 역할을 파악해 요약해준다. 처음 보는 코드베이스를 파악하거나, 팀 프로젝트에 뒤늦게 합류했을 때 특히 유용하다.

### Step 3: 파일 읽기와 요약

특정 파일의 내용을 이해하고 싶다면 읽어달라고 요청하면 된다.

```
README.md를 읽고 이 프로젝트가 무엇인지 한 문단으로 요약해줘
```

Claude가 파일을 열어 내용을 분석하고 요약문을 작성한다. 파일 이름만 알려주면 프로젝트 루트에서 알아서 찾는다.

### Step 4: 파일 생성

이제 실제 파일을 만들어보자.

```
새 파일 hello.py를 만들어줘. "Hello, Claude Code!"를 출력하는 간단한 스크립트야.
```

Claude가 파일 생성을 제안하면, 터미널에 아래와 같은 권한 프롬프트가 나타난다.

```
Claude wants to write to: hello.py

  print("Hello, Claude Code!")

Allow? [Y/n/d/?]
```

`Y`를 입력하면 파일이 생성된다. 이후 실제로 파일이 만들어졌는지 확인한다.

```bash
python hello.py
# Hello, Claude Code!
```

### Step 5: 권한 프롬프트 이해하기

파일 수정이나 셸 명령 실행 전에 Claude는 반드시 사용자 승인을 요청한다. 이것이 Claude Code의 핵심 안전 장치다.

프롬프트에서 선택지는 다음과 같다.

- `Y` 또는 Enter: 이번 한 번만 허용 (Allow Once)
- `a`: 이 세션에서 같은 종류의 작업을 항상 허용 (Allow Always)
- `n`: 거부 (Deny)
- `d`: 상세 정보 보기

처음에는 `Y`로 하나씩 확인하면서 Claude가 어떤 작업을 하는지 파악하는 습관을 들이자.

## 권한 모델 이해하기

Claude Code의 권한 모델은 에이전트가 임의로 시스템을 변경하지 못하게 설계됐다. 파일 쓰기, 파일 삭제, 셸 명령 실행 모두 사용자 승인 없이는 진행되지 않는다.

**Allow Once**는 해당 작업 하나에만 적용된다. 같은 종류의 작업이 다시 필요하면 다시 묻는다. 낯선 코드베이스에서 작업하거나 처음 시도하는 작업이라면 이 방식을 권장한다.

**Allow Always**는 세션 동안 같은 종류의 작업을 자동으로 허용한다. 신뢰할 수 있는 반복 작업, 예를 들어 테스트 실행이나 빌드 명령에는 편리하다.

자주 사용하는 도구를 영구적으로 허용하려면 프로젝트 루트의 `.claude/settings.json`에서 설정하면 된다.

```json
{
  "allowedTools": [
    "Read",
    "Write",
    "Bash(npm run test:*)",
    "Bash(git status)",
    "Bash(git diff)"
  ]
}
```

위 예시는 파일 읽기와 쓰기, 테스트 실행, 일부 git 명령을 사전 승인한 것이다. 패턴 매칭을 지원하므로 `npm run test:*`처럼 특정 명령 prefix만 허용하는 세밀한 제어도 가능하다.

`allowedTools`에 등록하지 않은 명령은 여전히 실행 전에 승인을 묻는다. 이 설계 덕분에 편의성과 안전성의 균형을 직접 조절할 수 있다.

## 실무 인사이트: 초보자가 흔히 하는 3가지 실수

Claude Code를 처음 쓰는 개발자들이 반복적으로 겪는 패턴이 있다. 미리 알아두면 시행착오를 줄일 수 있다.

### 실수 1: 권한 프롬프트를 무조건 수락한다

빠르게 작업하고 싶어서 권한 프롬프트가 뜰 때마다 반사적으로 `Y`를 누르는 경우가 많다. 그러나 이는 위험할 수 있다.

예를 들어 Claude가 오래된 파일을 정리한다는 의도로 `rm -rf ./build` 명령을 제안할 수 있다. 대부분의 경우 문제없지만, 맥락에 따라 의도하지 않은 디렉토리가 삭제될 수도 있다. 파일 생성보다 삭제나 셸 명령 실행 프롬프트에서는 반드시 내용을 읽고 판단하는 습관이 필요하다.

특히 `rm`, `git reset`, `DROP TABLE` 같은 돌이킬 수 없는 작업이 포함된 프롬프트는 항상 한 번 더 확인하자.

### 실수 2: 프로젝트 루트가 아닌 위치에서 실행한다

`~/Documents`나 홈 디렉토리에서 `claude`를 실행한 뒤, 작업 지시 안에 파일 경로를 명시하는 방식으로 사용하려는 경우가 있다.

```bash
# 잘못된 방식
cd ~
claude
# "~/my-project/src/app.py를 수정해줘"
```

이 방식이 동작할 수도 있지만, Claude가 프로젝트의 전체 구조, 패키지 설정, 린트 설정 등을 파악하지 못한다. 결과적으로 프로젝트 컨벤션에 맞지 않는 코드를 생성하거나, 다른 파일과의 의존 관계를 놓치게 된다.

항상 작업 대상 프로젝트의 루트에서 Claude를 실행하자.

```bash
# 올바른 방식
cd ~/my-project
claude
```

### 실수 3: 너무 큰 작업을 한 번에 요청한다

"이 프로젝트에 사용자 인증 기능을 추가해줘"처럼 범위가 큰 요청을 한 번에 넣으면 결과 품질이 떨어지기 쉽다. Claude가 추측해야 할 항목이 많아져 엉뚱한 방향으로 흐를 수 있다.

대신 작업을 단계별로 나눠 요청하면 훨씬 좋은 결과를 얻는다.

```
# 한 번에 요청 (비추천)
"이 프로젝트에 JWT 기반 사용자 인증 기능을 추가해줘"

# 단계별 요청 (추천)
1. "현재 프로젝트에 어떤 인증 관련 코드가 있는지 파악해줘"
2. "JWT 라이브러리를 설치하고 기본 설정 파일을 만들어줘"
3. "로그인 엔드포인트를 구현해줘. 이메일/비밀번호 방식으로"
4. "토큰 검증 미들웨어를 추가해줘"
```

각 단계마다 결과를 확인하고 방향을 조정할 수 있어, 마지막에 전체를 뒤엎는 상황을 피할 수 있다.
