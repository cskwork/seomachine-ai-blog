매일 반복하는 코드 리뷰 요청, 테스트 실행 지시, 배포 전 체크리스트 확인. Claude Code를 쓴다면 이런 반복 작업을 매번 긴 프롬프트로 입력하고 있을 가능성이 높습니다. 커스텀 슬래시 커맨드는 이 문제를 정면으로 해결합니다. 한 번 정의해두면 `/review`, `/test-feature`, `/deploy-check` 같은 짧은 명령 하나로 복잡한 워크플로우를 실행할 수 있다.

이 글에서는 커스텀 슬래시 커맨드의 작동 원리부터 시작해, 실무에서 바로 쓸 수 있는 세 가지 예제를 전체 코드와 함께 소개합니다.

---

## 슬래시 커맨드란

Claude Code의 슬래시 커맨드는 `.claude/commands/` 디렉토리 안에 마크다운 파일로 정의하는 재사용 가능한 프롬프트 템플릿입니다. 파일 이름이 곧 커맨드 이름이 된다. `review.md`라는 파일을 만들면 `/review`로 실행할 수 있다.

작동 방식은 단순하다. 마크다운 파일의 내용 전체가 Claude에게 전달되는 프롬프트가 된다. 매번 긴 지시문을 타이핑하는 대신, 한 번 잘 작성한 프롬프트를 파일로 저장해두고 반복 호출하는 구조다.

사용자 입력이 필요한 경우 `$ARGUMENTS` 변수를 사용합니다. `/review src/auth.ts`처럼 커맨드 뒤에 인자를 붙이면, 프롬프트 내의 `$ARGUMENTS`가 `src/auth.ts`로 치환됩니다.

```
슬래시 커맨드 작동 흐름:
/review src/auth.ts
  → .claude/commands/review.md 로드
  → $ARGUMENTS = "src/auth.ts" 로 치환
  → 완성된 프롬프트를 Claude에게 전달
  → Claude가 해당 파일에 대해 코드 리뷰 수행
```

## 기본 커맨드 만들기

처음 커스텀 커맨드를 만드는 과정은 세 단계다.

**Step 1: 디렉토리 생성**

프로젝트 루트에 `.claude/commands/` 디렉토리를 만듭니다.

```bash
mkdir -p .claude/commands
```

**Step 2: 마크다운 파일 작성**

첫 번째 커맨드로 간단한 `hello.md`를 작성해보자.

```markdown
# .claude/commands/hello.md

사용자에게 친근하게 인사하고, 오늘 작업할 내용이 있는지 물어봐줘.
만약 인자가 전달됐다면 ($ARGUMENTS), 그 내용을 언급하면서 인사해줘.
```

**Step 3: 실행 테스트**

Claude Code 세션에서 `/hello`를 입력하면 커맨드가 실행됩니다. `/hello 새로운 기능 개발`처럼 인자를 붙여도 됩니다.

이것만으로 기본 커스텀 커맨드는 완성입니다. 이제 실무에서 진짜 유용한 예제로 넘어갑니다.

## 실전 예제 1: /review - PR 리뷰 자동화

코드 리뷰는 개발자가 Claude에게 가장 자주 요청하는 작업 중 하나다. 매번 "버그는 없는지 확인해줘", "보안 취약점을 봐줘" 같은 지시를 나열하는 대신, 일관된 리뷰 기준을 파일로 정의해두면 팀 전체가 같은 품질 기준으로 리뷰받을 수 있다.

```markdown
# .claude/commands/review.md

아래 변경된 파일에 대해 코드 리뷰를 수행해줘.

리뷰 대상: $ARGUMENTS

다음 네 가지 관점에서 분석하고, 각 이슈마다 심각도를 표시해줘:

**1. 버그 가능성**
- 엣지 케이스 누락 여부
- 널 참조, 배열 범위 초과, 타입 불일치 가능성
- 비동기 처리의 race condition 위험

**2. 보안 취약점**
- SQL 인젝션, XSS, CSRF 취약점
- 하드코딩된 비밀번호나 API 키
- 인증/인가 로직의 허점

**3. 성능 문제**
- 불필요한 재렌더링이나 반복 연산
- N+1 쿼리 패턴
- 메모리 누수 가능성

**4. 코딩 컨벤션 위반**
- 프로젝트 네이밍 컨벤션 준수 여부
- 함수 길이와 복잡도
- 중복 코드와 추상화 기회

심각도 기준:
- Critical: 즉시 수정 필요 (서비스 장애, 보안 침해 가능)
- High: 이번 PR에서 반드시 수정
- Medium: 다음 스프린트 내 수정 권장
- Low: 개선 제안 (선택 사항)

리뷰가 끝나면 전체 요약과 함께 "수정 없이 머지 가능 여부"를 판단해줘.
```

사용법은 간단하다.

```bash
# 특정 파일 리뷰
/review src/auth/login.ts

# 여러 파일 리뷰
/review src/auth/login.ts src/auth/middleware.ts

# 변경된 파일 전체 리뷰 (git diff 기반)
/review $(git diff --name-only HEAD~1)
```

`$ARGUMENTS`에 파일 경로를 전달하면 Claude가 해당 파일의 내용을 읽고 정의된 기준에 맞게 리뷰를 수행합니다.

## 실전 예제 2: /test-feature - 기능 테스트 워크플로우

새 기능을 구현한 후 "테스트 잘 됐나요?"라는 질문에 자신 있게 대답하려면 체계적인 테스트 확인이 필요합니다. `/test-feature` 커맨드는 기능명을 인자로 받아 관련 테스트 파일을 찾고, 커버리지를 확인하고, 누락된 테스트 케이스를 제안하는 전체 워크플로우를 자동화합니다.

```markdown
# .claude/commands/test-feature.md

기능명: $ARGUMENTS

위 기능과 관련된 테스트를 전방위적으로 점검해줘. 다음 순서로 진행해:

## 1단계: 관련 파일 탐색

다음 패턴으로 관련 파일을 찾아줘:
- 소스 파일: `src/**/*$ARGUMENTS*`, `src/**/$ARGUMENTS/**`
- 테스트 파일: `tests/**/*$ARGUMENTS*`, `**/__tests__/*$ARGUMENTS*`, `**/*.test.*`, `**/*.spec.*`

찾은 파일 목록을 정리해서 보여줘.

## 2단계: 테스트 실행

발견된 테스트 파일에 대해 실행 명령을 제안해줘:
- Jest 프로젝트: `npx jest --testPathPattern="$ARGUMENTS" --coverage`
- Vitest 프로젝트: `npx vitest run --reporter=verbose $ARGUMENTS`
- pytest 프로젝트: `pytest tests/ -k "$ARGUMENTS" -v --cov`

프로젝트의 package.json이나 pyproject.toml을 확인해서 적합한 명령을 골라줘.

## 3단계: 커버리지 분석

현재 테스트가 다루고 있는 케이스와 다루지 않는 케이스를 분류해줘:

**커버되는 케이스:**
- (현재 테스트에서 검증하는 시나리오 목록)

**커버되지 않는 케이스:**
- (누락된 엣지 케이스, 오류 시나리오, 경계값)

## 4단계: 누락 테스트 케이스 제안

커버되지 않는 케이스에 대해 구체적인 테스트 코드를 작성해줘. 프로젝트에서 사용 중인 테스트 프레임워크 스타일로 맞춰줘.

각 테스트에는 다음을 포함해:
- 명확한 테스트 설명 (it/test 블록의 설명)
- Arrange-Act-Assert 구조
- 예상 결과값
```

사용 예시:

```bash
# 로그인 기능 테스트 점검
/test-feature login

# 결제 모듈 테스트 점검
/test-feature payment

# 사용자 프로필 기능 테스트 점검
/test-feature user-profile
```

이 커맨드의 핵심은 단순히 "테스트를 실행해줘"가 아니라, 테스트 완성도를 능동적으로 평가하고 개선 방향까지 제시한다는 점입니다.

## 실전 예제 3: /deploy-check - 배포 전 체크리스트

배포 직전에 "혹시 빠뜨린 게 없나?"라는 불안감은 모든 개발자가 겪습니다. `/deploy-check` 커맨드는 이 불안감을 체계적인 점검으로 해소합니다. 빌드 성공 여부, 테스트 통과, 타입 오류, 환경 변수 누락까지 배포 전에 반드시 확인해야 할 항목을 빠짐없이 검토합니다.

```markdown
# .claude/commands/deploy-check.md

배포 환경: $ARGUMENTS

배포 전 최종 점검을 수행해줘. 각 항목을 순서대로 확인하고 결과를 체크리스트 형태로 보고해줘.

## 체크리스트

### 빌드 및 컴파일
- [ ] 빌드 명령이 오류 없이 완료되는지 확인
  - 확인 명령: `npm run build` 또는 `yarn build`
  - 오류가 있다면 오류 내용과 수정 방법을 알려줘

### 테스트 통과
- [ ] 전체 테스트 스위트가 통과하는지 확인
  - 확인 명령: `npm test` 또는 `yarn test`
  - 실패한 테스트가 있다면 목록과 원인을 알려줘

### 타입 안전성
- [ ] TypeScript 타입 오류가 없는지 확인
  - 확인 명령: `npx tsc --noEmit`
  - 타입 오류가 있다면 위치와 수정 방법을 알려줘

### 코드 품질
- [ ] 린트 오류가 없는지 확인
  - 확인 명령: `npm run lint` 또는 `npx eslint src/`
  - 오류가 있다면 자동 수정 가능 여부와 수동 수정 필요 항목을 구분해줘

### 환경 변수
- [ ] 필요한 환경 변수가 모두 설정되어 있는지 확인
  - `.env.example` 또는 환경 변수 문서를 참고해서 누락된 변수를 찾아줘
  - 프로덕션 환경($ARGUMENTS)에 맞는 값이 설정됐는지 확인

### 의존성
- [ ] `package.json`의 의존성과 `node_modules`가 동기화되어 있는지 확인
  - `package-lock.json`이나 `yarn.lock`의 변경사항이 커밋됐는지 확인

### 보안
- [ ] 하드코딩된 비밀값이 없는지 확인
  - API 키, 비밀번호, 토큰이 소스 코드에 직접 포함되지 않았는지 점검
  - `.env` 파일이 `.gitignore`에 포함되어 있는지 확인

## 최종 판정

모든 항목 점검 후 다음 중 하나로 결론 내려줘:
- **배포 가능**: 모든 항목이 통과됨
- **조건부 배포 가능**: Minor 이슈가 있으나 배포 차단 수준이 아님 (이슈 목록 포함)
- **배포 불가**: 반드시 수정이 필요한 Critical 이슈가 있음 (이슈 목록과 수정 우선순위 포함)
```

사용 예시:

```bash
# 프로덕션 배포 전 점검
/deploy-check production

# 스테이징 배포 전 점검
/deploy-check staging
```

환경 이름을 인자로 전달하면 환경별 특성을 고려해 점검할 수 있다.

## 프로젝트 커맨드 vs 사용자 커맨드

커스텀 커맨드를 저장할 수 있는 위치는 두 곳입니다.

**프로젝트 커맨드: `.claude/commands/`**

프로젝트 루트의 `.claude/commands/` 디렉토리에 저장합니다. 이 파일들은 Git으로 추적되므로 팀 전체가 공유합니다. `/review`, `/deploy-check`처럼 팀 공통 워크플로우에 적합하다. 새 팀원이 합류하면 저장소를 클론하는 것만으로 팀의 모든 커스텀 커맨드를 바로 쓸 수 있다.

**사용자 커맨드: `~/.claude/commands/`**

홈 디렉토리의 `~/.claude/commands/`에 저장합니다. 모든 프로젝트에서 쓸 수 있는 전역 개인 커맨드다. 팀에 공유하기 어려운 개인 선호나 개인 워크플로우에 적합하다.

**우선순위**

같은 이름의 커맨드가 두 위치에 모두 존재하면 프로젝트 커맨드가 우선합니다. 덕분에 팀 공통 기준을 프로젝트 커맨드로 정의하면서도, 개인 커맨드로 기본 동작을 재정의할 수 있다.

```
우선순위: 프로젝트(.claude/commands/) > 사용자(~/.claude/commands/)
```

## 실무 인사이트: 커맨드 설계 원칙

커스텀 커맨드를 실무에서 효과적으로 활용하려면 몇 가지 설계 원칙을 지키는 것이 좋다.

**단일 책임 원칙**

하나의 커맨드는 하나의 명확한 목적만 가져야 한다. `/review-and-test-and-deploy`처럼 모든 것을 한 커맨드에 넣으면 재사용성이 떨어지고 유지보수가 어렵다. `/review`, `/test-feature`, `/deploy-check`처럼 목적별로 분리하면 각각을 독립적으로 사용하거나 순서를 바꿔 조합할 수 있다.

**에이전트 호출을 포함한 복합 워크플로우**

커맨드 안에서 다른 전문 에이전트를 호출하도록 설계할 수 있습니다. 예를 들어 `/security-review` 커맨드가 보안 전문 에이전트를 실행하도록 지시하면, 단순한 프롬프트 템플릿을 넘어 본격적인 워크플로우 자동화가 가능해집니다.

**$ARGUMENTS 활용 전략**

`$ARGUMENTS`는 커맨드를 범용적으로 만드는 핵심 도구입니다. 파일 경로, 기능명, 환경 이름처럼 상황마다 달라지는 값을 인자로 받으면, 하나의 커맨드 파일로 수십 가지 상황을 처리할 수 있습니다. 인자 없이도 동작하도록 기본 동작을 정의해두면 더 유연합니다.

**팀 표준화**

커스텀 커맨드의 가장 큰 가치 중 하나는 팀 내 워크플로우 표준화입니다. 프로젝트 커맨드로 팀 공통 기준을 정의하면, 신입 개발자도 처음부터 팀 기준에 맞는 리뷰, 테스트, 배포 프로세스를 따를 수 있습니다. 온보딩 비용이 줄고 코드 품질의 일관성이 높아집니다.

커맨드 파일 자체도 코드와 마찬가지로 리뷰하고 개선해야 합니다. 팀에서 자주 쓰는 커맨드는 정기적으로 내용을 점검해 더 정확하고 유용한 프롬프트로 발전시키는 것이 좋습니다.
