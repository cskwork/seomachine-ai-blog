## 디자인에서 프로덕션까지, 핸드오프 없이 가능할까

디자인 완성 후 개발 핸드오프 과정은 오래전부터 병목 지점이었다. 스펙 문서 작성, 개발자 해석, 리뷰, 피드백 루프를 거치면 원래 의도와 달라진 "그럭저럭 비슷한" 결과물이 몇 주 만에 나옵니다.

Figma MCP(Model Context Protocol)와 Claude Code의 조합은 이 구조를 근본적으로 바꾼다. Claude가 Figma 디자인 파일을 직접 읽고, 프로덕션 수준의 코드를 생성하는 워크플로우다.

> 이 글은 Felix Lee의 ["How to Build with Figma MCP"](https://adplist.substack.com/p/how-to-build-with-figma-mcp-the-designers) 원문을 참고하여 한국어로 재구성한 가이드입니다.

### 기존 워크플로우 vs 바이브 코딩

| 단계 | 기존 방식 | 바이브 코딩 |
|------|----------|------------|
| 1 | 디자인 완성 | 디자인 완성 |
| 2 | 스펙 문서 작성 | Claude가 Figma 직접 분석 |
| 3 | 개발자 해석 | "이거 빌드해줘" 프롬프트 |
| 4 | 리뷰 + 피드백 반복 | 프로덕션 코드 생성 |
| 5 | "비슷하게" 완성 (수 주) | 배포 (수 분) |

## Figma MCP 설정 방법

### 1단계: Figma 개인 액세스 토큰 생성

1. [Figma.com](https://figma.com) 로그인
2. 프로필 설정 > Security 이동
3. Personal Access Token 생성
4. 토큰을 즉시 복사해 안전한 곳에 저장

### 2단계: Claude Code에 연결

터미널에서 다음 명령어를 실행한다:

```bash
claude mcp add figma -e FIGMA_PERSONAL_ACCESS_TOKEN=figd_xxxxxxxxxx
```

`figd_xxxxxxxxxx` 부분을 1단계에서 복사한 실제 토큰으로 교체합니다.

### 3단계: 연결 확인

Claude Code에서 `/mcp` 명령어를 입력합니다. `figma` 항목이 connected 상태로 표시되면 설정 완료다.

## Claude가 Figma에서 읽을 수 있는 것과 없는 것

MCP를 통해 Claude는 Figma 파일의 구조적 데이터에 접근합니다. 다만 모든 정보를 읽을 수 있는 건 아니다. 읽을 수 있는 범위를 먼저 파악합시다.

### 읽을 수 있는 정보

- 레이어 이름과 계층 구조
- 색상값과 색상 스타일
- 타이포그래피 스타일 (폰트, 크기, 행간)
- Auto Layout 설정 (방향, 간격, 패딩)
- 컴포넌트와 Variant
- 간격과 제약 조건(Constraints)
- 텍스트 콘텐츠
- 디자인 토큰
- 이펙트 (그림자, 블러 등)

### 읽을 수 없는 정보

- 이미지 픽셀 데이터 (래스터 이미지 내용)
- 프로토타입 인터랙션
- 코멘트
- 버전 히스토리

이미지 에셋은 별도로 내보내기(export)해서 프로젝트에 추가해야 합니다.

## 4단계 구현 워크플로우

Figma 디자인을 코드로 변환할 때 다음 4단계를 순서대로 진행합니다.

### Phase 1: 디자인 구조 분석

Claude에게 Figma 파일 URL을 전달하고 전체 구조 분석을 요청합니다. Claude가 파악하는 항목은 다음과 같다:

- 페이지 섹션 구분
- 사용된 색상 팔레트
- 타이포그래피 체계
- 컴포넌트 목록

### Phase 2: 디자인 토큰 추출

분석 결과를 바탕으로 디자인 토큰을 추출합니다. Tailwind CSS를 사용한다면, 시맨틱 색상 이름이 포함된 Tailwind 설정 파일로 변환할 수 있습니다.

```
예: primary-blue → --color-primary: #2563EB
    body-text → font-size: 16px, line-height: 1.6
```

### Phase 3: 컴포넌트 생성

추출한 토큰을 기반으로 React(또는 지정한 프레임워크) 컴포넌트를 생성합니다. Figma의 스타일과 정확히 일치하는 TypeScript 컴포넌트가 만들어집니다.

### Phase 4: 섹션 조립

개별 컴포넌트를 조합해 전체 페이지 섹션을 구성합니다. 시맨틱 HTML 구조와 반응형 레이아웃까지 함께 적용합니다.

## 3가지 주요 활용 사례

### 1. Figma 디자인 파일에서 코드 변환

가장 일반적인 사용 패턴입니다. 완성된 디자인 파일의 URL을 Claude에게 전달하고, 원하는 기술 스택을 지정하면 프로덕션 코드를 바로 얻을 수 있습니다.

### 2. FigJam 와이어프레임에서 앱 구현

FigJam으로 그린 사용자 플로우나 와이어프레임도 변환 가능하다. 정밀한 디자인이 아닌 흐름 중심의 빠른 프로토타이핑에 유용하다.

### 3. Figma Make로 기존 코드베이스에 통합

Figma Make 기능을 활용하면 AI가 생성한 UI를 기존 프로젝트에 직접 주입할 수 있습니다. 가장 빠른 통합 방식입니다.

## 디자인 파일 준비 체크리스트

Figma MCP의 코드 품질은 디자인 파일 정리 수준에 달려 있습니다. 작업 전 다음 항목을 점검합시다.

- [ ] **시맨틱 레이어 이름 사용** - "Group 5" 대신 "nav-links", "hero-title" 등 의미 있는 이름
- [ ] **모든 컨테이너에 Auto Layout 적용** - 프레임 구조가 명확해야 코드 변환이 정확
- [ ] **색상을 스타일로 정의** - 하드코딩된 컬러값 대신 Color Style 사용
- [ ] **타이포그래피를 텍스트 스타일로 정의** - 일관된 폰트 체계
- [ ] **일관된 간격 시스템** - 8px 배수 등 규칙적인 spacing
- [ ] **계층 구조 단순화** - 불필요한 중첩 최소화

## 실전 프롬프트 팁

### 기본 원칙

1. **레이어 이름을 CSS 클래스처럼 짓는다** - Claude가 레이어 이름을 그대로 클래스명으로 사용하므로, 처음부터 코드 친화적인 네이밍이 중요하다.

2. **섹션 단위로 작업한다** - 전체 페이지를 한 번에 요청하지 말고, Hero > Navigation > Features > Footer 순서로 나눠서 요청합니다.

3. **Figma URL을 직접 참조한다** - "이 디자인을 보고" 대신 실제 Figma 파일 링크를 프롬프트에 포함합니다.

4. **이미지 에셋은 미리 내보낸다** - Claude는 이미지 픽셀을 읽지 못하므로, 필요한 이미지는 export 후 프로젝트 폴더에 배치합니다.

5. **기술 스택을 먼저 알려준다** - Next.js, Tailwind, TypeScript 등 사용할 스택을 프롬프트 시작 부분에 명시합니다.

### 퀵 스타트 프롬프트 템플릿

```
이 Figma 디자인을 분석하고 코드로 변환해줘:
[Figma URL 붙여넣기]

기술 스택: Next.js 14 + Tailwind CSS + TypeScript

다음 순서로 진행해줘:
1. 전체 디자인 구조 분석
2. 디자인 토큰 추출 → Tailwind 설정 생성
3. 섹션별 컴포넌트 생성
4. 반응형 레이아웃 적용
```

## 주의사항과 한계

Figma MCP는 강력하지만 만능은 아니다.

- **복잡한 애니메이션**은 별도 구현이 필요하다
- **이미지 에셋**은 수동 내보내기가 필수다
- **프로토타입 인터랙션**은 읽히지 않으므로 별도 설명이 필요하다
- **디자인 파일 정리 수준**이 코드 품질을 결정합니다. 정리되지 않은 파일은 정리되지 않은 코드를 만든다

---

## 정리

Figma MCP + Claude Code 조합의 핵심은 디자이너와 개발자 사이의 번역 과정을 없애는 데 있습니다. 디자인 의도가 코드로 곧바로 변환되니, "원래 디자인이랑 다른데요"라는 피드백 루프가 사라집니다.

잘 정리된 Figma 파일 + 명확한 프롬프트 = 빠르고 정확한 프로덕션 코드. 이것이 바이브 코딩의 핵심 공식입니다.
