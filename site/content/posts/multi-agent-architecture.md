## 왜 멀티 에이전트인가

단일 LLM 에이전트는 다음과 같은 한계에 직면합니다:

- **컨텍스트 윈도우 제한**: 모든 정보를 하나의 프롬프트에 담을 수 없음
- **전문성 분산**: 하나의 프롬프트가 모든 역할을 수행하면 품질 저하
- **단일 실패 지점**: 하나의 에이전트가 실패하면 전체 작업 중단
- **확장성 한계**: 작업 복잡도 증가에 따른 성능 저하

멀티 에이전트 시스템은 이러한 한계를 전문화와 분업으로 해결합니다.

## 아키텍처 패턴

### 1. Orchestrator-Worker 패턴

하나의 오케스트레이터가 전체 작업을 관리하고, 전문 워커 에이전트에게 하위 작업을 위임합니다.

```
사용자 요청
    ↓
[오케스트레이터]
    ├── [코드 분석 워커]
    ├── [테스트 작성 워커]
    ├── [문서화 워커]
    └── [보안 검토 워커]
    ↓
통합된 결과
```

장점:
- 중앙 집중적 제어로 일관성 유지
- 워커 간 독립적 실행 가능
- 새로운 워커 추가가 용이

단점:
- 오케스트레이터가 병목이 될 수 있음
- 오케스트레이터의 판단 오류가 전체에 영향

적합한 경우:
- 대규모 코드 리팩토링
- 종합적인 문서 생성
- 복합적인 분석 작업

### 2. Pipeline 패턴

에이전트가 순차적으로 작업을 처리하며, 각 단계의 출력이 다음 단계의 입력이 됩니다.

```
[리서치] → [초안 작성] → [편집] → [SEO 최적화] → [최종 검토]
```

장점:
- 각 단계가 명확하게 분리
- 중간 결과를 검증할 수 있음
- 디버깅이 용이

단점:
- 순차 실행으로 속도가 느림
- 앞 단계의 오류가 전파

적합한 경우:
- 콘텐츠 제작 파이프라인
- 데이터 처리 워크플로우
- 코드 리뷰 프로세스

### 3. 계층형 패턴

여러 레벨의 에이전트가 계층적으로 구성됩니다.

```
[프로젝트 매니저]
    ├── [프론트엔드 리드]
    │       ├── [React 개발자]
    │       └── [CSS 전문가]
    └── [백엔드 리드]
            ├── [API 개발자]
            └── [DB 전문가]
```

장점:
- 복잡한 프로젝트를 체계적으로 관리
- 각 레벨에서 독립적 의사결정
- 실제 팀 구조를 반영

단점:
- 구조가 복잡해질 수 있음
- 계층 간 통신 오버헤드

### 4. 토론/합의 패턴

여러 에이전트가 동일한 문제에 대해 독립적으로 분석하고, 토론을 통해 합의에 도달합니다.

장점:
- 다양한 관점에서 문제를 분석
- 편향 감소
- 높은 품질의 의사결정

단점:
- 비용이 높음 (여러 모델 호출)
- 합의에 도달하지 못할 위험

적합한 경우:
- 아키텍처 의사결정
- 보안 감사
- 중요한 코드 리뷰

## 설계 원칙

### 1. 단일 책임

각 에이전트는 하나의 명확한 역할만 수행해야 합니다.

- 나쁜 예: "코드를 분석하고 리팩토링하고 테스트도 작성하는 에이전트"
- 좋은 예: "코드 품질 분석 에이전트", "리팩토링 에이전트", "테스트 작성 에이전트"

### 2. 명확한 인터페이스

에이전트 간 통신 형식을 표준화합니다.

- 입력/출력 스키마 정의
- 에러 처리 규약
- 상태 보고 형식

### 3. 실패 복원력

개별 에이전트의 실패가 전체 시스템을 중단시키지 않도록 합니다.

- 타임아웃 설정
- 재시도 로직
- 폴백 전략
- 부분 결과 활용

### 4. 관찰 가능성

각 에이전트의 동작을 추적하고 디버깅할 수 있어야 합니다.

- 모든 에이전트 호출 로깅
- 에이전트 간 메시지 추적
- 성능 메트릭 수집
- 비용 모니터링

## 실무 구현 시 고려사항

### 모델 선택

모든 에이전트에 동일한 모델을 사용할 필요가 없습니다:

- 오케스트레이터: 고성능 모델 (Opus, o3)
- 전문 워커: 중간 성능 모델 (Sonnet, GPT-4o)
- 단순 작업: 경량 모델 (Haiku, mini)

### 컨텍스트 관리

에이전트 간 전달되는 컨텍스트를 최소화합니다:

- 필요한 정보만 선택적으로 전달
- 요약을 활용하여 컨텍스트 압축
- 공유 메모리(Redis, DB)를 통한 상태 관리

### 비용 관리

멀티 에이전트는 단일 에이전트 대비 비용이 높습니다:

- 에이전트 호출 횟수 제한
- 모델 라우팅으로 비용 최적화
- 캐싱으로 중복 호출 방지

## 시작하기

1. **단일 에이전트부터**: 먼저 하나의 에이전트로 핵심 기능을 구현
2. **병목 식별**: 단일 에이전트의 한계가 드러나는 지점 파악
3. **분리**: 해당 부분을 별도 에이전트로 분리
4. **통합**: 오케스트레이션 레이어 추가
5. **최적화**: 모니터링 기반으로 성능과 비용 최적화
